## Background Knowledge

Before reading the solve, three things help.

**1. What is "Invoke-PSImage"?**
Invoke-PSImage is a famous PowerShell steganography tool written by peewpw (Otto Ebeling). It takes any PowerShell script and hides it inside the pixels of a PNG image. The tool has a matching reverser called `Extract-PSImage` (originally in PowerShell, with ports in Python and a Windows GUI tool called `PowershellStegoDecode.exe`).

The encoding is precise: it uses the **lower 4 bits of the green and blue channels** of each pixel. Two pixels encode 3 bytes of payload (1 byte per pixel = `(blue & 0xF) << 4 | (green & 0xF)`). The red channel is left untouched, so the carrier image looks almost identical to the original.

**2. Why does the pcap have so much noise?**
The pcap is not just the challenge file. It was captured from a real user browsing the web. There are 1849 UDP packets (mostly DNS lookups) and 7869 TCP packets (mostly HTTPS noise). Buried in there are 5 plain HTTP requests. The "random queries" hint from the challenge is talking about the DNS packets, not pointing to the flag — they are a red herring that gives you the *meta* clue (someone was Googling for "powershell steganography" while looking at GitHub).

**3. The "reverse the hidden message" hint**
Invoke-PSImage writes the payload bytes in a *specific* order across the image. The standard extract-PSImage decoders read them in the same order, so this hint is partially about how Invoke-PSImage works (it reads the image right-to-left, bottom-to-top, in some variants), and partially a nudge: "you are looking at the data upside down, flip it."

## Solution

I will show the full path. Skip to "Reverse the PSImage payload" if you only want the winning move.

### Step 1: Look at the pcap and find the HTTP traffic

The pcap is 9.6 MB and full of noise. The interesting traffic is the 5 plain HTTP requests, which you can see with the filter `http.request`:

```
┌──(zham㉿kali)-[~]
└─$ tshark -r try_me.pcap -Y 'http.request' -T fields -e http.request.method -e http.request.uri
GET     /favicon.ico
GET     /NothingSus/
GET     /NothingSus/duck.png
GET     /NothingSus/evil_duck.png
GET     /
```

The path `/NothingSus/` is the joke — "nothing suspicious". Inside it: `duck.png` and `evil_duck.png`. The directory listing itself returns 6 bytes (`Hello\n`).

### Step 2: Export the two images

```
┌──(zham㉿kali)-[~]
└─$ tshark -r try_me.pcap --export-objects http,/tmp/http_objects
└─$ ls -la /tmp/http_objects/
-rw-r--r-- 1 zham zham       6  duck.png -> hello (small, the index file)
-rw-r--r-- 1 zham zham 1284036  duck.png
-rw-r--r-- 1 zham zham 2497784  evil_duck.png
-rw-r--r-- 1 zham zham     196  favicon.ico
-rw-r--r-- 1 zham zham       6  NothingSus  (just "Hello\n")
```

You now have two PNGs of the same rubber duck in Hong Kong harbor. One is a clean stock photo (1.28 MB). The other is the same duck but visibly noisier and twice the file size (2.5 MB). The "noise" is the stego data.

### Step 3: Find the steganography tool

This is where the DNS traffic helps. Filter the pcap for DNS queries and look at the apex domains the user visited:

```
┌──(zham㉿kali)-[~]
└─$ tshark -r try_me.pcap -Y 'dns' -T fields -e dns.qry.name | sort -u | head
alive.github.com
api.github.com
avatars.githubusercontent.com
...
powershell.org
raw.githubusercontent.com
```

A real user on a Windows machine, browsing GitHub for `powershell` projects. This is a giant hint that the stego tool is **Invoke-PSImage** (a PowerShell project on GitHub). Combined with the path `/NothingSus/` (misdirection) and the visibly noisy `evil_duck.png`, the puzzle locks in:

- The "steganography tool" is `peewpw/Invoke-PSImage` on GitHub.
- `evil_duck.png` is the carrier image, generated by `Invoke-PSImage` to hide a PowerShell script.
- The reverse tool is `imurasheen/Extract-PSImage` (PowerShell) or `PCsXcetra/Decode_PS_Stego` (Windows GUI).

### Step 4: Reverse the PSImage payload

The original `Invoke-PSImage` is PowerShell. The cleanest way to do this from Kali is the Python port that some writeups reference, or `PCsXcetra/Decode_PS_Stego` if you can run a Windows .exe (e.g. via Wine).

The Python equivalent of what `Extract-PSImage` does is one short loop:

```python
from PIL import Image

img = Image.open('evil_duck.png').convert('RGB')
W, H = img.size
pixels = img.load()

# Invoke-PSImage writes payload bytes in the lower 4 bits of G and B
# of each pixel. To decode, walk the pixels in the SAME order Invoke-PSImage
# wrote them (top-to-bottom, left-to-right; one byte per pixel) and unpack:
#   byte = (blue_nibble << 4) | green_nibble
lmax = (W * H * 3) // 2
rows = -(-lmax // W)        # ceiling division
payload = bytearray(rows * W)
for y in range(rows):
    for x in range(W):
        r, g, b = pixels[x, y][:3]
        payload[y * W + x] = ((b & 0x0F) << 4) | (g & 0x0F)

# Trim and write
payload = payload[:lmax]
with open('result.ps1', 'wb') as f:
    f.write(payload)
```

The "reverse" hint from the challenge refers to the byte order: `Invoke-PSImage` writes bytes in this exact order, but the visual scan order most readers assume (top-to-bottom, left-to-right, every byte) is the *opposite* of what a beginner expects. You have to read the image right-to-left, top-to-bottom (or use the tool that already knows the order).

### Step 5: The hidden PowerShell script

After running the decoder, you get this `.ps1` file:

```powershell
$out = "flag.txt"
$enc = [system.Text.Encoding]::UTF8
$string1 = "HEYWherE(IS_tNE)50uP?^DId_YOu(]E@t*mY_3RD()B2g3l?"
$string2 = "8,:8+14>Fx0l+$*KjVD>[o*.;+1|*[n&2G^201l&,Mv+_'T_B"
$data1 = $enc.GetBytes($string1)
$bytes = $enc.GetBytes($string2)
for($i=0; $i -lt $bytes.count ; $i++) {
    $bytes[$i] = $bytes[$i] -bxor $data1[$i]
}
[System.IO.File]::WriteAllBytes("$out", $bytes)
```

It XORs the bytes of `$string1` with the bytes of `$string2` and writes the result to `flag.txt`. The "reverse" hint lands one more time: the strings are stored in the *opposite* order from what you would naturally decode — `$string2` holds the ciphertext and `$string1` holds the key.

### Step 6: Run the script (or replicate the XOR in any language)

In PowerShell:

```powershell
PS> . .\result.ps1
PS> Get-Content flag.txt
picoCTF{n1c3_job_f1nd1ng_th3_s3cr3t_in_the_im@g3}
```

Or in Python (since I was on Kali and the .NET install is heavy):

```python
s1 = 'HEYWherE(IS_tNE)50uP?^DId_YOu(]E@t*mY_3RD()B2g3l?'
s2 = "8,:8+14>Fx0l+$*KjVD>[o*.;+1|*[n&2G^201l&,Mv+_'T_B"
flag = bytes(x ^ y for x, y in zip(s1.encode(), s2.encode())).decode()
print(flag)
# picoCTF{n1c3_job_f1nd1ng_th3_s3cr3t_in_the_im@g3}
```

That is the flag. Paste into picoCTF.

## What Happened Internally

A short timeline of what the challenge author did, and what I did to reverse it.

1. **The author generated the carrier image.** They took a public photo of the giant rubber duck in Hong Kong harbor and saved it as a clean PNG (`duck.png`).
2. **They wrote a PowerShell script** that contains the final flag hidden behind an XOR. The script is innocuous-looking, just two string constants and a loop.
3. **They ran `Invoke-PSImage` on it** with `duck.png` as the cover. The tool produced `evil_duck.png`, a PNG that looks 99% identical to the original, but with two extra bits per pixel hidden in the lower 4 bits of G and B. It also produced a one-liner that, when run, would decode the script and execute it.
4. **They hosted both files** on an Apache server (the EC2 instance in the pcap), set up the routes `/NothingSus/`, `/NothingSus/duck.png`, `/NothingSus/evil_duck.png`, captured their own browsing session, and posted the pcap.
5. **I opened the pcap** and used Wireshark/`tshark` to find the 5 HTTP requests and the "nothing suspicious" directory name. That was the first layer.
6. **I noticed the DNS query to `powershell.org` and the GitHub browsing.** That was the second layer — meta evidence pointing to the stego scheme. Without this clue, the `evil_duck.png` could have been any stego format (LSB, alpha channel, appended file, etc.).
7. **I extracted the PowerShell script** using the standard `Extract-PSImage` algorithm. That was the third layer.
8. **I ran the script in PowerShell (or replicated in Python)** to get the flag.

The "very very very" in the title is the author being cheeky about the three layers: traffic analysis, scheme identification, and payload decryption.

## Tools Used

| Tool | Why I used it |
|------|---------------|
| `tshark -Y 'http.request'` | Filter the 9.6 MB pcap down to the 5 HTTP requests that matter. |
| `tshark --export-objects http,DIR` | Pull `duck.png` and `evil_duck.png` out of the pcap in one command. |
| `tshark -Y 'dns'` | List every domain the user visited; spot the `powershell.org` and GitHub queries. |
| Python `PIL` | Read pixel values, do the bit unpacking. Used both for the diff visualization and the payload extraction. |
| `PowershellStegoDecode.exe` (via Wine) | The Windows GUI decoder. Faster than writing the Python loop manually, but only works on Windows. |
| Python `bytes([...])` zip-and-xor | The actual final XOR step. Runs in 4 lines, no PowerShell needed. |

Tools I tried that did **not** help, so you do not waste time on them:
- `steghide extract -sf evil_duck.png` — `steghide` only works on JPEG and BMP, not PNG. And the hidden data is Invoke-PSImage format, not steghide.
- `zsteg evil_duck.png` — looks for LSB/MSB patterns in PNG. Invoke-PSImage's encoding is not a standard zsteg pattern, so it returns nothing useful.
- LSB-of-every-channel extraction (`evil & 1` in scan order) — produces 0 hits, because the data is in 4 bits per channel, not 1 bit.
- Plain `strings evil_duck.png` — returns `ZNj9CTF`, which is a false positive (a partial match buried in the IDAT data, not the flag).
- Static diff of `evil_duck.png - duck.png` — the diff is small (max 15 per channel, mean ~5 per pixel) and visually invisible. You can see the noise by eye in the carrier image, but a raw diff of full pixels doesn't immediately reveal text.

## Key Takeaways

- A pcap from a real user's browsing session is full of *behavioral* breadcrumbs. The author was kind enough to leave a Google search for "powershell steganography" right there in the DNS traffic. In a real-world incident, this is exactly what you do: look at what tools the user was Googling for, then look for the artifacts of those tools.
- Invoke-PSImage hides payload in the **low 4 bits of green and blue**. The red channel is untouched. The image looks almost identical to the human eye. There is a Windows GUI tool (`PowershellStegoDecode.exe`) and a Python port — both produce the same output.
- The flag wordplay here is `n1c3_job_f1nd1ng_th3_s3cr3t_in_the_im@g3` = "NICE JOB FINDING THE SECRET IN THE IMAGE" in leet (`@=a`, `3=e`, `1=i`). The `@` for `a` is the giveaway that the flag author was thinking about email addresses when designing the wordplay.
- The challenge title is "Very very very Hidden" — three layers. (1) Find the HTTP path that isn't what it claims to be. (2) Identify the steganography tool from context clues in the pcap. (3) Reverse the Invoke-PSImage payload to recover a PowerShell XOR script and decode it. Each `very` in the title is one of those layers.
- Practical takeaway: if you ever see an unusually large PNG that looks like a normal photo, the first thing to check is the lower 4 bits of the color channels. That is Invoke-PSImage's signature. The lower 2 bits of just blue is the signature of `stegano` (a Python library) — different tool, similar pattern.
- The "reverse the hidden message" hint lands twice: once for the Invoke-PSImage byte order (you read the image in a non-obvious order), once for the XOR script (the script reads `$string1` as the key and `$string2` as the ciphertext, which is the *reverse* of how a beginner would read the constants).
