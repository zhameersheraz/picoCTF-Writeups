## Background Knowledge (Read This First!)

### TLS 1.2 with static RSA key exchange

This pcap is a single TLS 1.2 connection to `172.31.22.220:443` from `128.237.140.23`. The server's chosen cipher is **`TLS_RSA_WITH_AES_256_GCM_SHA384` (0x009d)**, which uses static RSA key exchange (not ephemeral Diffie-Hellman). With static RSA, the pre-master secret is generated by the client, encrypted with the server's public RSA key, and sent in the `ClientKeyExchange` message. Anyone with the server's private key can decrypt the pre-master secret and recover the session keys.

### Why this matters for forensics

If you have the server's private RSA key, you do NOT need to be a MITM to decrypt the TLS session after the fact. The static RSA key exchange makes the session completely retroactively decryptable: pull the pre-master secret out of the pcap, decrypt it with the private key, derive the master secret and key block, and decrypt the Application Data records with AES-256-GCM. This is exactly what Wireshark's "RSA keys list" feature does when you add the key file to the TLS preferences.

### Why the flag is in a JPEG, not the HTTP headers

The description "The flag has moved — it is no longer in the HTTP headers" hints that the natural place to look (decrypt the TLS, follow the HTTP stream, read the headers) gives a **decoy**. The HTTP response contains a `Pico-Flag:` header that reads `picoCTF{this.is.not.your.flag.anymore}` — a red herring. The real flag is embedded as a plain ASCII string inside the JPEG image (`vulture.jpg`) that the server sends in the body of another response. Once you have the JPEG bytes, `strings vulture.jpg | grep pico` finds it immediately.

## Solution — Step by Step

### Step 1 — Confirm the setup

```
┌──(zham㉿kali)-[/media/sf_downloads]
└─$ file capture.pcap picopico.key
capture.pcap:  pcap capture file, microsecond ts (little-endian) - version 2.4
picopico.key:  PEM RSA private key, 2048-bit
```

```
┌──(zham㉿kali)-[/media/sf_downloads]
└─$ openssl rsa -in picopico.key -text -noout | head -2
Private-Key: (2048 bit, 2 primes)
modulus:
    00:b0:2a:51:4f:34:a8:ec:78:91:79:a6:e0:89:53:
```

A 2048-bit RSA key. The pcap has 110 packets, 1 main TLS flow to `172.31.22.220:443`.

### Step 2 — Open the pcap in Wireshark and load the key

```
Wireshark → Edit → Preferences → Protocols → TLS → RSA keys list
  IP address:    172.31.22.220
  Port:          443
  Protocol:      http
  Key File:      /path/to/picopico.key
  Key:           (leave blank — key is read from file)
```

Click OK and re-open the pcap. The "Application Data" packets turn green, which means Wireshark successfully decrypted the AES-256-GCM records using the session keys derived from your private key.

### Step 3 — Follow the HTTP stream

`Right-click any decrypted packet → Follow → TLS Stream` (or `HTTP Stream`).

The decrypted stream looks roughly like this:

```
GET / HTTP/1.1
Host: ec2-18-223-184-200.us-east-2.compute.amazonaws.com
...

HTTP/1.1 200 OK
Pico-Flag: picoCTF{this.is.not.your.flag.anymore}
Content-Type: text/html
...
```

The `Pico-Flag` header is a decoy — `picoCTF{this.is.not.your.flag.anymore}`. The description hinted that the flag has moved, so keep digging.

### Step 4 — Export the HTTP objects

`File → Export Objects → HTTP`. You'll see several files transferred, including a `vulture.jpg`. Save it.

Equivalent tshark command:
```
tshark -r capture.pcap \
  -o 'ssl.keys_list:0.0.0.0,443,http,picopico.key' \
  --export-objects 'http,./out'
```

### Step 5 — Find the flag inside the JPEG

The flag is embedded as a plain ASCII string inside the JPEG's binary data. `strings` finds it immediately:

```
┌──(zham㉿kali)-[/media/sf_downloads/out]
└─$ strings vulture.jpg | grep pico
picoCTF{honey.roasted.peanuts}
```

The recovered flag is `picoCTF{honey.roasted.peanuts}`.

## What Happened Internally (Timeline)

1. The challenge author set up a TLS 1.2 server on `172.31.22.220:443` using a self-signed certificate. The corresponding private RSA key was saved as `picopico.key` and given to the player.
2. The client (a scripted browser?) connected, negotiated `TLS_RSA_WITH_AES_256_GCM_SHA384`, and exchanged session keys via static RSA. This means the session's master secret is recoverable from the pcap by anyone holding the private key.
3. The server responded with several HTTP resources: an HTML page (with the decoy Pico-Flag header) and a JPEG file (`vulture.jpg`).
4. The real flag was written into the JPEG's binary data as a plain ASCII string. The image is a vulture photo that the author found amusing — the wordplay `honey.roasted.peanuts` is a nod to "Honey Badger Don't Care" internet meme culture (also a reference to a popular cybersecurity YouTube channel).
5. You, the solver, take the private key, load it into Wireshark, decrypt the TLS session, see the decoy header, export the JPEG, and grep for the flag inside the image bytes.

## Key Takeaways

- **Static RSA key exchange = retroactive decryption.** With ephemeral (DHE/ECDHE) key exchange, even the server's private key cannot recover the session — that's the whole point of forward secrecy. Static RSA (any cipher starting with `TLS_RSA_WITH_*`) does not have this property. When you see a static RSA cipher in a pcap, the key file in the challenge is a decryption oracle, not just a distraction.
- **A flag in a header is the easy part.** The challenge author specifically says "the flag has moved" — they put a decoy in the obvious place (HTTP header) and hid the real one in the binary payload of an image. Don't stop at the first `picoCTF{...}` you see.
- **`strings` is still king for embedded strings.** Even without any image-analysis tool, `strings` finds the flag because it was stored as plain ASCII bytes inside the JPEG data. Always run `strings` on extracted files before reaching for specialized stego tools.
- **The wordplay is `honey.roasted.peanuts`.** The vulture image + peanuts is a reference to a well-known internet meme ("Honey Badger Don't Care"), not anything specific to the cryptography in the challenge. The flag is the punchline.
- **`ssldump` works the same way as Wireshark for this challenge.** If you don't have a GUI, `ssldump -r capture.pcap -k picopico.key -d > output` decrypts the stream and dumps the plaintext to a file. Same end result, different path.
